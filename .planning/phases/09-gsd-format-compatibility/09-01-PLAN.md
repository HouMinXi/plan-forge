    ## Reference Class

    Three comparable multi-dynamic-gate projects with plan-vs-actual implementation
    ratios and lessons:

    - Chromium CQ tiering (2018-present): diff-based CI tier selection (which builders
      to run based on affected files). Integration: estimated ~1.5x planned. The
      directory-to-builder mapping was straightforward; shared-header impact radius was
      the long tail. Lesson: shared dependencies are the hard case, not leaf components.

    - Rust bors / pre-merge queue (2015-present): serialized merge queue that runs the
      full test suite before merging. Pre-merge execution is the correct design for
      preventing regressions; the tradeoff is queue serialization. Ratio: ~1.2x planned
      (estimated). Lesson: pre-merge gates are load-bearing; post-merge revert-first
      policies are the escape hatch, not the primary design.

    - Forge Phase 1 (internal reference): commit gate + install-hooks. Four plans,
      estimated 5 days, actual 7 days including reviews = ratio 1.4x. Main overrun:
      EC-9 review found 11 additional findings; the install_hooks backup-conflict bug

    ## Pre-mortem

    Imagine the v2.1 milestone ships and the dynamic gates fail to work in practice.
    The most likely causes:

    1. R1 lands but the gate is silently bypassed in CI via FORGE_MODE or env vars.
       Early warning: CI builds show no gate-check output in logs; a deliberately-red
       tree is merged without blocking. Counter: CI detection is SPEC-locked (FORGE_MODE
       + platform vars); FORGE_SKIP_TESTS is explicitly rejected in CI mode.

    2. R2 lands but equivalent mutants flood the review with CONFIRMED findings that
       engineers cannot action. Early warning: review loops never converge because a
       new MUTANT survivor appears every round. Counter: consecutive_survivor_rounds
       hard-stops at 3; MUTANT findings are CONFIRMED (human triage), not EXIT_FAIL
       (not a commit blocker).

    3. R3 components.yaml is too painful to author for code/kernel/networking (106
       subsystems). Early warning: the first real components.yaml takes >1 day to
       write. Counter: auto-detection workflow documented; Layer 1 is non-blocking and
       still fires without components.yaml.

    4. Phase 0 baseline becomes stale after tests are added or removed; gate-check
       blocks clean trees. Early warning: gate-check blocks a commit that adds a new
       test. Counter: baseline is regenerated by re-running Phase 0 bootstrap; the
       schema documents the regeneration command.

    5. The three-cycle review itself becomes the bottleneck: review takes longer than
       implementation. Early warning: a single 15-line fix triggers 1h+ review without
       convergence. Counter: single-model review converges what a multi-model panel
       spins on (memory feedback_multimodel_review_small_change_nonconvergence.md);
       circuit-break at 3 non-converging rounds.

    ## Scope Challenge

    Q1: Does this need to exist?
    Yes. Forge ships the thesis "verification grounding beats prompt-only self-claim"
    while its own commit gate ran no tests (marker-trusting blind spot). The specific
    cost of doing nothing: CLAUDE.md honest assessment section explicitly states "until
    R2 and R3 land, forge is partially implementing its own thesis." This is a
    self-contradiction that is visible to every user who reads the docs.

    Q2: Three real consumers
    - Forge itself: dogfood -- forge's own commits are gated by R1, mutation-checked
      by R2, and cross-component-checked by R3.
    - plan-forge: uses the forge review pipeline; R2 adds mutation coverage on
      plan-forge's own changed files.
    - Any project running forge review with a .forge/gate.yaml: gets the real commit
      gate and mutation pipeline without additional configuration.

    Q3: Do-nothing cost
    Doing nothing + documenting the gap: forge continues to ship the verification
    grounding thesis while its own commit gate relies on a self-claim marker. The
    honest assessment in CLAUDE.md documents this explicitly. The cost is reputational:
    a tool that argues verification grounding > self-claim while not applying its own
    standard to itself.

    Q4: Barbell vs middle ground
    Barbell options: (A) static review only, no dynamic gates (status quo, documented
    gap); (B) full automated test generation + mutation + full e2e call-graph analysis
    (years of work, unsolved for shell/C). R1+R2+R3 is NOT the middle ground -- it IS
    the minimal viable dynamic gate set that closes the specific blind spot (no-test
    commit gate) without the unsolved problems (call-graph extraction, cross-repo
    impact). Function-level mutation scoping and cross-repo impact are incremental
    enhancements.

    ## External Voices

    Primary sources on dynamic gate design and mutation testing in practice:

    - Petrovic & Ivankovic (2018). State of Mutation Testing at Google. ICSE 2018.
      Documents Google's deployment of mutation testing to ~6,000 engineers who author
      or review code, affecting over 13,000 code authors (roughly 30% of all diffs).
      Key findings: diff-scoped mutation is the enabling design decision for CI
      practicality; results integrated into code review workflow, not a separate async
      gate; engineer acceptance requires framing surviving mutants as actionable
      suggestions, not failures. Primary real-world reference for the diff-scoped design
      in forge R2.

    - Jia & Harman (2011). An Analysis and Survey of the Development of Mutation
      Testing. IEEE Transactions on Software Engineering, Vol. 37, No. 5. The
      most-cited comprehensive survey on mutation testing. Documents the long-standing
      adoption challenge: equivalent mutants produce false positives, making
      mutation-adequate tests hard to achieve in practice.

    A legitimate objection exists: the Jia & Harman survey documents that equivalent
    mutants are a fundamental unsolved problem. A surviving mutant may indicate a test
    weakness OR an equivalent mutant -- the reviewer cannot distinguish them
    automatically. Critics argue this reduces actionability. Forge mitigates by
    classifying survivors as CONFIRMED (review-worthy, not blocking commits) rather
    than hard CI failures.

    Historical lesson from Petrovic & Ivankovic (2018): early internal pilots of
    mutation at Google without diff-scoping produced runs too slow to be actionable.
    The retrospective lesson: selective (diff-scoped) mutation was the enabling design
    decision. Forge R2 inherits this constraint as a first-principles design
    requirement, not an optimization.
