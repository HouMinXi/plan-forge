## Cycle 1

### Pass 1 (qodo-review)

**C1P1-F1 [MEDIUM] YAML frontmatter comment line causes premature break**

Location: _own_id_set, frontmatter loop, around line 84

    if stripped.startswith("#") or stripped == "---":
        break

A YAML comment inside frontmatter (e.g. "# Generated by GSD") matches
startswith("#") and breaks out of the loop before reading phase: or
plan: fields that follow. Result: own ID is not extracted, and body
self-references are incorrectly reported as F3 findings.

Trigger: user or tool writes a "#" comment before phase: in frontmatter.
Current GSD orchestrator does not generate such comments, so the risk is
low in practice. Fix: skip (continue) YAML comment lines inside the loop
instead of breaking on them.

**C1P1-F2 [LOW] heading scan does not strip BOM from heading lines**

Location: _own_id_set, heading scan loop

    if line.strip().startswith("#"):

str.strip() does not remove U+FEFF. A file with a BOM-prefixed heading
on its first line would not be recognised, causing the heading scan to
skip it. In practice, BOM appears only at file start and a heading is not
the first line when frontmatter is present, so impact is theoretical.

**C1P2-F3 [LOW] Malformed frontmatter (no closing ---) causes over-scanning**

Without a closing ---, the loop continues through the entire document body.
Any prose line matching "plan:\s*\d+" would be misread as a frontmatter
field, producing a spurious own ID. This is a GIGO case -- the document is
structurally invalid. Existing code is acceptable.

**C1P3-F4 [LOW] heading scan terminates prematurely on frontmatter comments**

Same root cause as F1. When frontmatter contains a "#" comment, the heading
scan loop mistakes it for the first ATX heading and breaks after extracting
nothing from it.
